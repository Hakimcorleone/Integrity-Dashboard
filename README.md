# Integrity Risk Modelling Lab

Integrity Risk Modelling Lab is an open-source prototype for integrity risk analytics using fully synthetic procurement and payment data. It demonstrates how rule-derived red-flag signals and machine learning can be combined to predict review priority levels for internal review planning, control testing, and monitoring workflows.

The project predicts an integrity risk level for review prioritisation. It does not detect, prove, confirm, or allege corruption, fraud, bribery, misconduct, or wrongdoing.

## Why This Project Exists

Integrity, audit, compliance, and procurement teams often need practical ways to prioritise large volumes of transactions for review. Real procurement and payment data can contain confidential, personal, commercial, and operationally sensitive information, making it unsuitable for public experimentation.

This project exists to provide a safe, synthetic, open-source lab for:

- prototyping integrity risk analytics workflows;
- learning how machine learning pipelines can support review prioritisation;
- demonstrating leakage-aware preprocessing and model comparison;
- building dashboards without using confidential or personal data;
- encouraging responsible discussion about analytics in integrity and internal control contexts.

## Key Features

- Generates 1,000 fully synthetic procurement/payment transactions.
- Creates weighted red-flag features and a capped `risk_score` from 0 to 100.
- Labels synthetic transactions as `Low`, `Medium`, `High`, or `Critical` review priority.
- Uses stratified train/test splitting for model evaluation.
- Avoids leakage by excluding identifiers, `risk_score`, and `review_required` from model features.
- Trains Logistic Regression, Decision Tree, Random Forest, and Gradient Boosting classifiers.
- Selects the best model using macro F1-score.
- Produces model comparison, classification report, confusion matrix, and feature-importance chart.
- Includes a Streamlit dashboard for dataset exploration, model results, and synthetic case prediction.

## Dataset Description

The dataset is generated by `src/generate_dataset.py` and saved to:

```text
data/raw/integrity_procurement_risk_dataset.csv
```

Each row represents one synthetic procurement/payment transaction. The dataset includes transaction attributes such as department, region, procurement method, contract and invoice amounts, claimed and verified units, bidder count, approval/checker flags, complaints, variation orders, delivery delay, duplicate invoice flag, round amount flag, and split purchase flag.

The generator also creates derived features such as:

- `invoice_contract_ratio`
- `invoice_contract_variance`
- `claim_variance_units`
- `claim_variance_pct`
- `direct_award_flag`
- `emergency_procurement_flag`
- `single_bidder_flag`
- `short_tender_flag`
- `missing_checker_flag`
- `missing_approver_flag`
- `complaint_flag`
- `vo_amount_ratio`
- `late_delivery_flag`

Risk levels are assigned from a synthetic score:

- `Low`: 0 to 24
- `Medium`: 25 to 49
- `High`: 50 to 74
- `Critical`: 75 to 100

Approximate generated distribution:

- Low: 40-50%
- Medium: 25-35%
- High: 15-20%
- Critical: 5-10%

All records, IDs, labels, and values are synthetic. Do not add real names, IC numbers, emails, phone numbers, bank account details, confidential data, or sensitive information to this repository.

## Machine Learning Approach

The modelling workflow uses `risk_level` as the target variable.

The preprocessing step excludes the following fields from model features to reduce leakage and avoid identifier-based learning:

- `transaction_id`
- `vendor_id`
- `risk_score`
- `review_required`

Categorical columns are one-hot encoded:

- `department`
- `region`
- `procurement_method`

The pipeline uses a stratified train/test split so the review-priority class distribution is preserved in both splits.

Models trained by `src/train_model.py`:

1. Logistic Regression
2. Decision Tree Classifier
3. Random Forest Classifier
4. Gradient Boosting Classifier

Evaluation metrics:

- accuracy
- macro precision
- macro recall
- macro F1-score
- weighted F1-score

The best model is selected using macro F1-score, which gives each class equal weight and is more useful than accuracy alone when reviewing performance across minority classes such as `Critical`.

## Limitations

- The dataset is synthetic and does not represent any real organisation, supplier, person, payment, or transaction.
- The risk labels are generated from rule-based logic, so model performance reflects the synthetic design rather than real-world complexity.
- The model should not be used for automated decisions or adverse actions.
- The prototype does not include fairness testing, drift monitoring, governance workflows, access controls, or human review case management.
- Feature importance values are model-specific signals and should not be interpreted as proof of any issue.
- Public deployments should not accept confidential, personal, commercially sensitive, or operationally sensitive data.

## Ethics and Responsible Use

Use this project as a learning and prototyping tool. Any real-world implementation of integrity analytics should include strong governance, legal review, data protection controls, human oversight, documentation, validation, and clear escalation procedures.

Responsible use principles:

- Use synthetic or properly authorised data only.
- Keep humans responsible for review decisions.
- Treat model outputs as prioritisation signals, not conclusions.
- Monitor false positives because they create unnecessary review workload.
- Monitor false negatives because they may miss cases that should be reviewed.
- Document assumptions, limitations, and model changes.
- Do not use the prototype to profile individuals or make employment, supplier, enforcement, disciplinary, or legal determinations.

## Disclaimer

“This tool does not determine whether corruption, fraud, bribery, misconduct, or wrongdoing has occurred. It is a red flag analytics and machine learning prototype intended to support preliminary review, internal control assessment, and integrity risk monitoring. All data is synthetic. Users should not upload confidential, personal, or sensitive information into public deployments.”
